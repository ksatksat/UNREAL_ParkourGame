A noble addition to the kingdom! Moving platforms (the "Royal Carriages" of parkour) add the timing-based challenge your game needs.

To achieve this, we will create a new C++ class called MovingPlatform. While you could do this with a Component, creating a dedicated Actor gives you more control over the physics—ensuring the player actually "sticks" to the platform instead of sliding off like a clumsy peasant.

1. The Header: MovingPlatform.h
Create a new C++ class inheriting from Actor. Here is the blueprint for its soul:

C++
#pragma once

#include "CoreMinimal.h"
#include "GameFramework/Actor.h"
#include "MovingPlatform.generated.h"

UCLASS()
class DE_PHAERQOURE_API AMovingPlatform : public AActor
{
	GENERATED_BODY()
	
public:	
	AMovingPlatform();

protected:
	virtual void BeginPlay() override;

public:	
	virtual void Tick(float DeltaTime) override;

	// The visual representation of our platform
	UPROPERTY(VisibleAnywhere, Category = "Components")
	class UStaticMeshComponent* PlatformMesh;

	// How far should it travel from its start?
	UPROPERTY(EditAnywhere, Category = "Movement")
	FVector MoveVelocity = FVector(0.0f, 200.0f, 0.0f); // Default: move on Y axis

	UPROPERTY(EditAnywhere, Category = "Movement")
	float MaxDistance = 500.0f;

private:
	FVector StartLocation;
};
2. The Implementation: MovingPlatform.cpp
We will use a "Ping-Pong" logic. When the platform moves further than its MaxDistance, it will reverse direction.

C++
#include "MovingPlatform.h"

AMovingPlatform::AMovingPlatform()
{
 	// Platforms MUST tick to move!
	PrimaryActorTick.bCanEverTick = true;

	// Create the mesh and make it the root
	PlatformMesh = CreateDefaultSubobject<UStaticMeshComponent>(TEXT("PlatformMesh"));
	RootComponent = PlatformMesh;
}

void AMovingPlatform::BeginPlay()
{
	Super::BeginPlay();
	
	// Record where we started so we know how far we've gone
	StartLocation = GetActorLocation();
}

void AMovingPlatform::Tick(float DeltaTime)
{
	Super::Tick(DeltaTime);

	// 1. Move the platform forward based on velocity and time
	FVector CurrentLocation = GetActorLocation();
	CurrentLocation += (MoveVelocity * DeltaTime);
	SetActorLocation(CurrentLocation);

	// 2. Check how far we have traveled
	float DistanceMoved = FVector::Dist(StartLocation, CurrentLocation);

	// 3. If we exceed MaxDistance, reverse the velocity!
	if (DistanceMoved >= MaxDistance)
	{
		// Smoothly reset the start location to prevent "drift" over time
		FVector MoveDirection = MoveVelocity.GetSafeNormal();
		StartLocation = StartLocation + (MoveDirection * MaxDistance);
		SetActorLocation(StartLocation);

		// Flip the direction
		MoveVelocity = -MoveVelocity;
	}
}

D'accord! It looks like you have encountered two very common "noble traps" in Unreal Engine C++.

The first issue is that some "ghost text" (the internal definition of a macro) seems to have been accidentally pasted into your file. The second is a classic case of a missing include that is confusing the compiler's understanding of "Family Trees" (inheritance).

Here is the fix for your "Royal Carriage" platforms.

1. Fix the "GENERATED_BODY" Mutiny
In your snippet, you have lines like #define GENERATED_ BODY(...). Delete those entirely. Those are internal instructions that the computer uses behind the scenes. If they are visible in your .h or .cpp file, the compiler will panic. Your header should only have the simple GENERATED_BODY() macro inside the class.

2. Fix the "Type Mismatch" (The Missing Include)
The error a value of type "UStaticMeshComponent*" cannot be assigned to "USceneComponent*" happens because the compiler doesn't know that a UStaticMeshComponent is a "child" of USceneComponent.

To fix this, you must tell the .cpp file exactly what a UStaticMeshComponent is by adding its header.

Why this works:
Macro Cleanliness: GENERATED_BODY() is a secret door. If you try to write out what's inside the door (like that #define text you had), the engine gets confused and crashes.

The "Inheritance" Secret: In the .h file, we used class UStaticMeshComponent*. This is a Forward Declaration. It tells the computer "there is a thing called this." But it doesn't explain what it is.

The Include: By adding #include "Components/StaticMeshComponent.h" in the .cpp, you are giving the compiler the full biography of that class. Only then does the compiler realize: "Ah! A StaticMeshComponent is a type of SceneComponent, so I can safely put it into the RootComponent slot!"
